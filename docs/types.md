# ducknng type support and schema policy

This document is binding for how `ducknng` declares and supports types in method inputs and outputs. The project does not want a vague promise that it uses Arrow. It needs an explicit, stable contract describing which Arrow logical types are accepted, how they map to DuckDB logical types, which subset is considered canonical for output, and which advanced or exotic types are intentionally deferred.

The guiding principle is that Arrow IPC is the transport-level tabular encoding and DuckDB logical types are the execution-level types. The protocol contract sits between them. That contract must be narrow enough to be reliable and broad enough to be useful. Early versions of the project should therefore prefer a strong and explicit supported subset over a theoretically complete but poorly specified compatibility claim.

The initial support target is the practical core of DuckDB tabular work. Boolean values, signed integers, unsigned integers, floating-point values, UTF-8 strings, binary values, dates, microsecond-resolution times, microsecond-resolution timestamps, structs, and lists should form the stable initial supported set. Those types cover a large share of realistic SQL and data interchange work without forcing the project into immediate support for the most complex corners of the Arrow type system.

DuckDB mappings should be documented and treated as normative. Arrow `bool` maps to DuckDB `BOOLEAN`. Arrow `int8`, `int16`, `int32`, and `int64` map to DuckDB `TINYINT`, `SMALLINT`, `INTEGER`, and `BIGINT`. Arrow `uint8`, `uint16`, `uint32`, and `uint64` map to DuckDB `UTINYINT`, `USMALLINT`, `UINTEGER`, and `UBIGINT`. Arrow `float32` and `float64` map to DuckDB `FLOAT` and `DOUBLE`. Arrow UTF-8 string values map to `VARCHAR`, binary values map to `BLOB`, Arrow `date32` maps to `DATE`, Arrow `time64[us]` maps to `TIME`, and Arrow `timestamp[us]` maps to `TIMESTAMP`. Arrow structs map to DuckDB `STRUCT`, and Arrow lists map to DuckDB list values or array-like nested types as supported through DuckDB’s logical type system.

The project should distinguish between what it accepts and what it emits. It is reasonable to accept a somewhat broader family of timestamp units or string widths than the server emits, but reply payloads should normalize toward a canonical output subset so clients receive predictable schemas. Canonical output is therefore part of the contract. At the current stage, the canonical output target should prefer ordinary UTF-8 strings rather than large UTF-8 unless size or implementation constraints require otherwise, and should prefer microsecond timestamps for temporal output unless a stronger preservation requirement is adopted deliberately.

For the current unary `exec(..., want_result = true)` implementation, the emitted row-result subset is intentionally narrower than the long-term target. The working unary row-reply subset is `BOOLEAN`, signed and unsigned integer types, `FLOAT`, `DOUBLE`, `VARCHAR`, and `BLOB`. Other result types should move to the future session-based query path or remain unsupported until their mapping is implemented and tested. This narrower current subset is acceptable because the protocol contract distinguishes between current concrete support and future intended coverage.

Nullability is part of the contract rather than an afterthought. Methods must declare whether fields are nullable, and implementations must not silently weaken or strengthen nullability without updating the manifest and this document. Since Arrow represents nullability explicitly and DuckDB also reasons about nulls directly, the server should preserve null semantics faithfully for supported types.

Nested types should begin with a disciplined subset. Structs and lists are in scope because they are immediately useful and fit well with Arrow and DuckDB alike. Deeply nested or unusual combinations are not forbidden in principle, but they should only be claimed as supported when tested and documented. The project should not casually imply support for arbitrary nested schemas until the implementation and test coverage justify that claim.

Several useful types belong in the near-term expansion tier rather than the strict initial core. Large UTF-8 strings, large binary values, decimal values such as `decimal128`, fixed-size binary values, duration types, and nanosecond timestamps are all reasonable next steps. Dictionary-encoded Arrow inputs may eventually be supported, but early support should prefer decoding them into plain values rather than promising dictionary-preserving roundtrips. That approach keeps interoperability practical while avoiding premature commitments to the trickier details of Arrow dictionary semantics.

Some Arrow features should be explicitly deferred. Unions, maps, run-end encoded arrays, extension types, dictionary-preserving fidelity guarantees, and other advanced encodings are not rejected forever, but they should not be advertised as supported until the server can describe and test them precisely. The project is better served by a narrow truthful contract than by a broad hand-wavy one.

Method schemas should declare concrete field names and concrete types rather than informal descriptions. For example, `exec` should declare a request schema containing a non-null `sql` UTF-8 field and a non-null `want_result` boolean field. Its metadata reply should declare concrete integer fields such as `rows_changed`, `statement_type`, and `result_type`. Methods that return dynamic SQL result sets should still describe whether those rows are returned directly or through session-based fetch behavior, even if the exact schema depends on the SQL query itself.

The session family needs explicit control schemas in addition to dynamic row schemas. `query_open` should declare an Arrow IPC request table with one logical row and at least these fields: non-null `sql: utf8`, nullable `batch_rows: int64`, nullable `batch_bytes: int64`, and nullable `args_json: utf8` if statement parameterization is introduced before a richer Arrow parameter table exists. Its JSON control reply should declare fields such as non-null `session_id: string`, non-null `state: string`, non-null `next_method: string`, nullable `schema_json: string`, non-null `initial_batch_inline: bool`, and nullable `warning: string`. `fetch`, `close`, and `cancel` should declare JSON request objects with non-null `session_id: string`; `fetch` may additionally declare nullable `batch_rows: int64` and `batch_bytes: int64` hints, while `close` and `cancel` may add nullable `reason: string` fields for auditability without changing semantics.

The dynamic row schema returned by `fetch` is the schema established by `query_open` for that one session and stays fixed for the life of the session. `fetch` must not change column names or logical types across batches of the same session. If the server needs to communicate only control state, such as end-of-stream, cancellation acknowledgement, or already-closed status, it returns JSON control metadata instead of an empty Arrow table with invented sentinel columns.

The manifest must expose enough type information for clients to generate or validate requests. This means the manifest should not merely say that a method takes Arrow IPC. It should say what Arrow fields are expected, whether fields are nullable, and what output modes are possible. The type contract therefore lives jointly in this document and in the manifest exported by the server.

Compatibility decisions must be explicit. If the server chooses to normalize `timestamp[ms]` input to `timestamp[us]` output, that is acceptable, but it must be written down. If the server accepts small and large string inputs but always emits ordinary UTF-8 strings, that too must be written down. The point is to make clients robust through clarity rather than surprise.

This document is also a checklist. Before a new method or new type claim is considered stable, the Arrow logical type must be named here, the DuckDB mapping must be declared here or in the manifest, nullability behavior must be clear, canonical output behavior must be clear, and test coverage must exist for both successful roundtrip behavior and failure behavior on unsupported types. If those conditions are not met, the type claim is not yet part of the public contract.
