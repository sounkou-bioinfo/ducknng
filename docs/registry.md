# ducknng method descriptor, registry, and dispatcher contract

This document makes the registry model in `docs/manifest.md` concrete enough to guide implementation. It is binding for the first registry-backed dispatcher. The goal is to ensure that the registry is not only a manifest source but also the enforcement point for method policy, security checks, payload limits, and future session ownership rules.

The first important rule is that every public RPC method must have a descriptor object and the dispatcher must resolve incoming calls through the registry rather than through hardcoded string comparisons. A handler that is reachable without a descriptor is outside the public contract. This rule applies even while the method set is small. It is better to have a registry that initially contains only `exec` and later `manifest` than to postpone the registry until many methods already exist in bespoke form.

The descriptor is the unit of policy. It must carry enough information for the dispatcher to decide whether a request is acceptable before the handler runs. That means the descriptor is not just metadata for documentation. It is also where request shape, session requirements, security posture, and resource limits are declared. The manifest is then derived from the same descriptor fields that the dispatcher enforces, so the client-visible contract and the runtime policy remain aligned.

The intended C shape is a `ducknng_method_descriptor` structure whose fields are stable in meaning even if their low-level representation evolves. The descriptor should include the method name, family, summary, transport pattern, request payload format, response payload format, response mode, and protocol version metadata already required by `docs/manifest.md`. It must also include a handler function pointer and the security and runtime policy fields needed for dispatch. Those policy fields should include whether the method requires authentication, whether it requires an existing session, whether it opens a session, whether it closes or cancels a session, whether it mutates server state, whether it is idempotent, whether it is deprecated but still enabled, whether it is fully disabled, the accepted request flag mask, the emitted reply flag mask, the maximum accepted request bytes, and the maximum permitted reply bytes.

A concrete first-pass C design should therefore look conceptually like a descriptor with immutable name and metadata strings, a small set of enum fields describing transport and payload behavior, a handler callback with a standard signature, and a policy block. The policy block should be plain data and not hidden inside handler-specific code. If a method is stateless, that fact should be declared in the descriptor. If it requires a session, that fact should be declared in the descriptor. If it is safe to expose pre-authentication, that fact should be declared in the descriptor. If it is capable of returning only small metadata replies, that fact should be declared in the descriptor. The dispatcher should not have to guess any of this from naming conventions.

The handler signature should also be standardized now. A method handler should receive the service pointer, the resolved descriptor, a request context object, and an output object for the reply. The request context should carry the decoded frame, any resolved caller identity once an identity model is chosen, any resolved session object if the dispatcher has already looked one up, and any request-scoped limits or tracing metadata that are relevant. The reply object should let the handler return either an error or a success payload without having to allocate a final NNG message manually. That keeps wire assembly centralized and makes later additions such as structured errors or tracing easier to apply uniformly.

The request context needs to become the place where security and ownership checks land. Even before a final authentication model is chosen, the design should reserve an identity field in the context so the dispatcher can later bind a caller identity derived from transport trust, TLS peer information, or RPC-level authentication. The same context should also carry a session token or resolved session pointer when a method references session state. This matters because the session family cannot be made safe if handlers are responsible for ad hoc ownership checks.

The registry itself should be represented by a dedicated `ducknng_method_registry` object owned by the service or runtime rather than by static global tables scattered across source files. The registry should support registration of one descriptor, lookup by method name, registration of a descriptor array in one call, and unregistration by name or by bundle or family token. Internally it may start as a simple array with linear lookup while the method set is small, but the public contract should not depend on the lookup algorithm. The important part is that lookup is authoritative and that registration and unregistration semantics are explicit.

Bulk registration must be a first-class feature rather than a helper around repeated single registration. The protocol is already organized around coherent method families, and the code should mirror that. A `core_control` family, a `core_exec` family, and a later `query_sessions` family should each be registerable through one bulk operation. Bulk unregistration must also have defined behavior. If a stateless family is unregistered, new calls to those methods simply stop resolving. If a sessionful family is unregistered, new session opens must stop immediately while existing sessions are drained or force-closed according to the documented policy in `docs/security.md`. This behavior belongs to the registry and session management contract rather than to one-off shutdown code.

The dispatcher should run through a fixed sequence. It first validates the envelope and the wire version. It then rejects unsupported message types and validates incoming flags against the descriptor-independent wire rules. After that it resolves the method name through the registry. Once the descriptor is found, the dispatcher validates that the incoming message type is appropriate for the descriptor, that the request payload length is within the descriptor's limit, and that the incoming flags are a subset of the descriptor's accepted request flag mask. Only then should it evaluate authentication requirements, resolve or validate session ownership, and finally invoke the handler. When the handler returns, the dispatcher validates that the reply flags are a subset of the descriptor's declared emitted reply flag mask and that the reply payload size is within the descriptor's maximum. This ordering is important because it keeps policy enforcement outside handlers.

Structured error handling should be part of the dispatcher contract as well. A handler should return a status code from the documented enum and may optionally supply a human-readable detail string, but the dispatcher should remain the place that turns that into a wire-level error reply. This keeps clients from depending on English error text and ensures that future additions such as machine-readable error classes or audit logging do not require every handler to reimplement reply construction.

The `manifest` method should eventually be generated directly from the registry's current descriptor state. That means the descriptor must contain enough data to describe schemas and policy meaningfully. Some schema material may live as JSON fragments, some may live as static strings or references to shared schema objects, but the registry must expose it uniformly enough that manifest generation is mechanical. A method that cannot be exported through the manifest is not ready to exist as a public method.

The first implementation milestone should be modest. It should introduce the descriptor type, the registry type, the request context and reply objects, and a dispatcher that resolves only `exec` through the registry. Even that limited milestone is valuable because it establishes the right architecture. Once it exists, adding `manifest` becomes a natural second step, and adding the session family later becomes an extension of an existing policy-aware dispatcher rather than a redesign under pressure.

The checklist implied by this document is straightforward. Before a method becomes public, it must have a descriptor, be present in the registry, expose its manifest-relevant metadata through that descriptor, declare its accepted request flags and emitted reply flags, declare its request and reply size limits, and use the standard handler signature. Before the dispatcher is considered correct, it must enforce the descriptor's policy before calling handlers and validate outgoing replies against the same descriptor. If either side of that relationship is missing, the registry is not yet serving its intended role as the public and security contract of the framework.
