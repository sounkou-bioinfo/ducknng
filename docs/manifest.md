# ducknng manifest and method registry

This document is binding for how `ducknng` methods are declared, registered, exposed, deprecated, and removed. The key rule is that the method registry is the server-side source of truth and the manifest is a generated or derived representation of that registry state. In other words, the project should not drift into a model where runtime behavior lives in one place and public capability description lives in another unrelated place.

The manifest exists to tell clients what methods exist, how those methods are named, which transport pattern they use, what payload formats they accept and emit, what schemas they expect and return, whether they are stateless or sessionful, whether they are experimental or deprecated, and which version of the protocol introduced them. The manifest is therefore not just descriptive documentation. It is part of the contract between the server and every client implementation.

The registry must support single-method operations and bulk operations. A single method must be registerable, unregisterable, and discoverable by name. Groups of methods must also be registerable and removable in bulk so that protocol families can be installed coherently instead of piecemeal. That is especially important for families such as core control, unary SQL execution, session queries, and future event-oriented protocols. The presence of bulk registration is not an implementation nicety; it is part of the intended architecture because the project explicitly wants a way to define, add, and remove methods as coherent sets.

Each public method needs a descriptor. The exact C layout may evolve, but the logical content is fixed. A descriptor must carry the method name, its family, a human-readable summary, its transport pattern, its request payload format, its response payload format, its response mode, its request schema declaration, its response schema declaration, its session behavior, whether it is idempotent, whether it is deprecated, and the protocol version in which it was introduced. Experimental status is still a project-level documentation concept for the session family today; once the registry and exported manifest grow an explicit experimental flag, that field should be added there instead of being implied only by prose.

Payload format values should remain small and explicit. A method may declare that it uses no payload, JSON payloads, or Arrow IPC stream payloads. Arrow IPC file payloads are reserved for future use if they ever become necessary, and custom binary payloads are strongly discouraged because they push the project back toward bespoke per-method framing. If a custom binary payload is ever proposed, it must justify why Arrow IPC or JSON is insufficient.

Response modes also need explicit declaration because they determine how a client interprets a successful reply. A method may return no payload, metadata only, rows, either metadata or rows depending on execution, session-opening information, or event-style output in the future. This distinction is especially important for `exec`, which can legitimately return either metadata about a non-row-producing statement or a tabular result when row output is requested.

Session behavior must be declared directly in the method descriptor rather than inferred from implementation details. A method can be stateless, open a session, require an already-open session, close a session, or cancel a session. That declaration matters because it controls both client expectations and how the server organizes its own execution state. A method that is logically sessionful but undocumented as such is a protocol defect.

The manifest should be exported as stable JSON because JSON is the easiest universal interchange format for discovery data across R, Python, Go, JavaScript, and C tooling. The manifest should contain server identity, protocol version information, service security metadata such as TLS auth mode, peer-identity policy, whether exact peer/IP allowlists are active, and whether a SQL authorizer is active, server-owned session policy metadata such as effective idle timeout, capability summaries, bundle or family grouping information, and a full list of method descriptors. If named schema reuse becomes useful later, the manifest may include shared schema references, but inline schema declarations are acceptable and probably clearer at the current stage.

Local SQL client helpers that do not name a server method are not manifest entries. This matters for the aio surface and for transport adapters. `ducknng_request_raw_aio(...)`, `ducknng_request_socket_raw_aio(...)`, `ducknng_get_rpc_manifest_raw_aio(...)`, `ducknng_run_rpc_raw_aio(...)`, `ducknng_send_socket_raw_aio(...)`, `ducknng_recv_socket_raw_aio(...)`, `ducknng_ncurl_aio(...)`, `ducknng_aio_ready(...)`, `ducknng_aio_collect(...)`, `ducknng_ncurl_aio_collect(...)`, `ducknng_aio_cancel(...)`, `ducknng_aio_drop(...)`, `ducknng_start_http_server(...)`, and `ducknng_ncurl(...)` are runtime-local client or transport helpers over ordinary declared methods, not registry methods exported by the server. The manifest should continue to describe the underlying methods and frame contracts, while those helpers remain local ergonomics layered on top.

The implemented HTTP/HTTPS carrier therefore does not create a second manifest namespace. The same descriptors for `manifest`, `exec`, `query_open`, `fetch`, `close`, and `cancel` remain authoritative regardless of whether the current carrier is NNG or HTTP/HTTPS. `docs/http.md` makes the same rule concrete by pinning a frame-over-HTTP carrier rather than a second method namespace.

For the session query family, the manifest must be concrete enough that a client can implement the full lifecycle without reading server source. `query_open` must declare that it opens exactly one query session, that the next expected method is `fetch`, that the reply payload is JSON control metadata containing `session_id`, `session_token`, and the server-owned effective `idle_timeout_ms`, and that the reply may later include an `initial_batch_inline` boolean plus an optional schema summary for the result stream. `fetch` must declare that it requires an open session, that it consumes both `session_id` and `session_token`, that it may emit either Arrow IPC row batches or JSON control metadata, and that its flags are where end-of-stream and any embedded-schema hints are communicated. `close` and `cancel` must each declare that they consume a `session_id` plus `session_token` JSON payload, that they do not emit row data, and whether they are idempotent, best-effort, or terminal.

A method schema declaration is normative for clients. That means the manifest should say not only that `exec` exists, but also that its request payload is an Arrow IPC stream containing a one-row table with fields such as `sql` and `want_result`, and that its reply is either an Arrow IPC metadata table or an Arrow IPC result table depending on execution. The same rule also applies to `query_open`, `fetch`, `close`, and `cancel`: `query_open` takes a one-row Arrow control table, while the other three use JSON control payloads keyed by `session_id` and `session_token`. Transport-derived mTLS identity is not serialized into those schemas; it is request context enforced by the dispatcher/session manager. If the manifest and the implementation disagree, the implementation should be considered wrong until the mismatch is resolved explicitly.

Method names should be grouped into stable families. The initial intended families are control, SQL, session, event, and experimental. Some names are implemented today and some are only reserved. In the current implementation `manifest` and the session family names `query_open`, `fetch`, `close`, and `cancel` are real manifest-visible methods, while `exec` remains an opt-in SQL family method rather than a guaranteed always-on baseline. Names such as `server_info` are reserved for the core protocol surface but should not be described as implemented until they actually exist in the registry and exported manifest. Reserving these names now avoids later confusion and keeps the main protocol easy to discover and reason about.

Adding a method requires a descriptor, registry integration, manifest exposure, schema declaration, and tests. Removing a method should normally proceed through deprecation unless the method was explicitly experimental. Deprecation itself must be visible in the manifest so clients can warn or adapt cleanly. This is especially important once external clients begin to rely on the server contract in a more stable way.

Bulk registration should be the normal way to install method families. The registry should be able to register a core control bundle, a core execution bundle, a session-query bundle, and future optional bundles such as experimental features or pub/sub event capabilities. This model makes it possible to enable or disable coherent surfaces rather than sprinkle individual methods throughout startup logic.

The manifest and registry together are the practical implementation checklist for the public RPC surface. A method should not be considered part of `ducknng` until its descriptor exists, the registry knows about it, the manifest exports it, its request and response schemas are declared, its session behavior is documented, and its versioning status is explicit. Descriptor `requires_auth` is enforced against verified transport-derived caller identity when present; listener policy may also require mTLS identity before any method dispatch. For the query family specifically, the manifest is incomplete unless it also states the session state transitions, `session_token` ownership requirement, server-owned effective idle timeout, optional mTLS owner-identity binding, idempotency of `close`, best-effort semantics of `cancel`, and the temporary REQ/REP constraint that all row delivery happens only on explicit `fetch` replies. That is the rule that keeps the project extensible without becoming shapeless.
