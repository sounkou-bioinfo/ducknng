# ducknng manifest and method registry

This document is binding for how `ducknng` methods are declared, registered, exposed, deprecated, and removed. The key rule is that the method registry is the server-side source of truth and the manifest is a generated or derived representation of that registry state. In other words, the project should not drift into a model where runtime behavior lives in one place and public capability description lives in another unrelated place.

The manifest exists to tell clients what methods exist, how those methods are named, which transport pattern they use, what payload formats they accept and emit, what schemas they expect and return, whether they are stateless or sessionful, whether they are experimental or deprecated, and which version of the protocol introduced them. The manifest is therefore not just descriptive documentation. It is part of the contract between the server and every client implementation.

The registry must support single-method operations and bulk operations. A single method must be registerable, unregisterable, and discoverable by name. Groups of methods must also be registerable and removable in bulk so that protocol families can be installed coherently instead of piecemeal. That is especially important for families such as core control, unary SQL execution, session queries, and future event-oriented protocols. The presence of bulk registration is not an implementation nicety; it is part of the intended architecture because the project explicitly wants a way to define, add, and remove methods as coherent sets.

Each public method needs a descriptor. The exact C layout may evolve, but the logical content is fixed. A descriptor must carry the method name, its family, a human-readable summary, its transport pattern, its request payload format, its response payload format, its response mode, its request schema declaration, its response schema declaration, its session behavior, whether it is idempotent, whether it is deprecated, whether it is experimental, and the protocol version in which it was introduced. Optional metadata can extend this, but the core contract cannot be omitted.

Payload format values should remain small and explicit. A method may declare that it uses no payload, JSON payloads, or Arrow IPC stream payloads. Arrow IPC file payloads are reserved for future use if they ever become necessary, and custom binary payloads are strongly discouraged because they push the project back toward bespoke per-method framing. If a custom binary payload is ever proposed, it must justify why Arrow IPC or JSON is insufficient.

Response modes also need explicit declaration because they determine how a client interprets a successful reply. A method may return no payload, metadata only, rows, either metadata or rows depending on execution, session-opening information, or event-style output in the future. This distinction is especially important for `exec`, which can legitimately return either metadata about a non-row-producing statement or a tabular result when row output is requested.

Session behavior must be declared directly in the method descriptor rather than inferred from implementation details. A method can be stateless, open a session, require an already-open session, close a session, or cancel a session. That declaration matters because it controls both client expectations and how the server organizes its own execution state. A method that is logically sessionful but undocumented as such is a protocol defect.

The manifest should be exported as stable JSON because JSON is the easiest universal interchange format for discovery data across R, Python, Go, JavaScript, and C tooling. The manifest should contain server identity and protocol version information, capability summaries, bundle or family grouping information, and a full list of method descriptors. If named schema reuse becomes useful later, the manifest may include shared schema references, but inline schema declarations are acceptable and probably clearer at the current stage.

A method schema declaration is normative for clients. That means the manifest should say not only that `exec` exists, but also that its request payload is an Arrow IPC stream containing a one-row table with fields such as `sql` and `want_result`, and that its reply is either an Arrow IPC metadata table or an Arrow IPC result table depending on execution. The same rule will later apply to `query_open`, `fetch`, `close`, and `cancel`. If the manifest and the implementation disagree, the implementation should be considered wrong until the mismatch is resolved explicitly.

Method names should be grouped into stable families. The initial intended families are control, SQL, session, event, and experimental. Certain names are reserved for the core protocol surface, including `manifest`, `server_info`, `exec`, `query_open`, `fetch`, `close`, and `cancel`. Reserving these names now avoids later confusion and keeps the main protocol easy to discover and reason about.

Adding a method requires a descriptor, registry integration, manifest exposure, schema declaration, and tests. Removing a method should normally proceed through deprecation unless the method was explicitly experimental. Deprecation itself must be visible in the manifest so clients can warn or adapt cleanly. This is especially important once external clients begin to rely on the server contract in a more stable way.

Bulk registration should be the normal way to install method families. The registry should be able to register a core control bundle, a core execution bundle, a session-query bundle, and future optional bundles such as experimental features or pub/sub event capabilities. This model makes it possible to enable or disable coherent surfaces rather than sprinkle individual methods throughout startup logic.

The manifest and registry together are the practical implementation checklist for the public RPC surface. A method should not be considered part of `ducknng` until its descriptor exists, the registry knows about it, the manifest exports it, its request and response schemas are declared, its session behavior is documented, and its versioning status is explicit. That is the rule that keeps the project extensible without becoming shapeless.
