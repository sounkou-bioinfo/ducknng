# ducknng subagent workflows

This repository includes project-local Pi skills intended to work well with a subagent-capable Pi setup. The local skills live under `.pi/skills/` and are meant to keep protocol, registry, session, security, and handoff work aligned with the binding repository contracts.

If the packaged subagent extension and agents are available in your Pi environment, the most relevant local skills are `ducknng-rpc-framework` and `ducknng-subagent-handoff`. The first is the architecture skill for protocol, registry, dispatcher, manifest, and session work. The second is the orchestration and handoff skill for splitting work across recon, planning, implementation, and review.

The best decomposition for this repo is by framework layer. A scout-style pass should focus on one layer at a time, such as wire framing, registry and dispatch, transport and security, or session lifecycle. A planner-style pass should turn the binding docs into a specific implementation sequence. A worker-style pass should then change only the code for that layer while keeping the docs authoritative. A reviewer-style pass should check that `docs/protocol.md`, `docs/manifest.md`, `docs/types.md`, `docs/security.md`, and `docs/registry.md` still match the implementation and that verification commands were actually run.

Useful subagent chains in this repo include scout to planner to worker for larger architectural changes, or parallel scouts when the work can be cleanly split between code and docs. Good examples are running one scout over the wire and registry paths and a second over README, tests, and generated artifacts, then merging their findings into a single implementation plan. Another good split is one scout for transport and security constraints and another for DuckDB execution and session state.

When using subagents here, every handoff should mention contract state, implementation state, and verification state. Contract state means which binding docs govern the changed area. Implementation state means whether the registry is authoritative yet, whether dispatch is fully registry-backed or hybrid, whether envelope versioning is enforced, and whether sessions have ownership. Verification state means which of `make release`, `make test_release`, `make rpc_smoke`, `make rpc_smoke_r`, and `make rdm` were run and what limitations remain. `make rpc_smoke` is the CI-safe sqllogictest path and must not depend on R. `make rpc_smoke_r` is the optional R/nanonext interop smoke path.

For session/query-family work, the handoff should also name the exact lifecycle decision set rather than saying only that "session docs were updated." State whether `query_open` returns JSON control metadata or Arrow rows, whether it returns `session_token`, whether `fetch` / `close` / `cancel` require that token, whether `fetch` is the only row-bearing method, whether `close` is idempotent, whether `cancel` is best-effort, what the documented lifecycle states are, and whether any remaining session limitations are about execution isolation rather than bare session ownership. That level of specificity is what prevents the next worker swarm from implementing mutually incompatible session semantics.

These workflows are intentionally repository-specific. They are meant to prevent the common failure mode where a subagent changes one part of the framework without updating the docs, tests, or generated artifacts that define the public contract.
