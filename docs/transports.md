# ducknng transport adapters and carrier boundaries

This document defines the transport-family boundary for `ducknng`. It is intentionally narrower than `docs/protocol.md`: the protocol document defines the manifest methods, session lifecycle, and payload rules, while this document defines how those same contracts are carried over concrete transport adapters.

The core rule is simple. Transport adapters may change how bytes move, but they do not change what the bytes mean. The `ducknng` envelope remains the method layer, Arrow IPC remains the tabular payload layer, JSON remains the control-metadata layer, and the manifest remains the source of truth for public RPC methods. A new transport adapter is therefore not a license to mint parallel method names or alternate session semantics.

Today the implemented transport family is NNG. URL schemes such as `inproc://`, `ipc://`, `tcp://`, and `tls+tcp://` all resolve into the NNG adapter layer, which is isolated in `src/ducknng_nng_compat.c`. The transport-family parser that classifies URL schemes lives separately in `src/ducknng_transport.c` so that future transport adapters can reuse the same endpoint classification instead of baking scheme logic into SQL glue or into the NNG shim itself.

The planned next family is HTTP and HTTPS. That future work is expected to surface through local SQL helpers such as `ducknng_start_http_server(...)` and `ducknng_ncurl(...)`, but those helpers are transport-local entry points, not new manifest methods. The manifest does not gain `http_exec`, `http_query_open`, `http_fetch`, or similar duplicates. It continues to describe `manifest`, `exec`, `query_open`, `fetch`, `close`, and `cancel` once, independent of whether the current carrier is NNG or a later HTTP adapter.

This matters most for sessions. Under an HTTP transport adapter, `query_open` still accepts the same one-row Arrow IPC control payload with `sql` and optional batch hints. `fetch` still accepts JSON control metadata keyed by `session_id`. `fetch` still returns either Arrow IPC row batches or JSON control metadata. `close` and `cancel` still consume JSON control payloads and still return JSON control replies. The session state machine is unchanged. Only the outer carrier changes.

The same rule holds for Arrow record batches. A future HTTP adapter should not reinterpret row batches as ad hoc JSON arrays or text serialization just because the outer transport is HTTP. When a method returns Arrow IPC today over NNG, it should continue to return Arrow IPC over HTTP. When a method returns JSON control metadata today over NNG, it should continue to return JSON control metadata over HTTP. Content headers may help a client understand the body, but they do not replace the method contract.

The intended code boundary is therefore three-way. Transport-family parsing belongs in transport-neutral code such as `ducknng_transport.c`. NNG-specific socket, listener, TLS, and aio details belong in `ducknng_nng_compat.c`. A future HTTP adapter should live in a separate HTTP-specific compatibility layer rather than extending the NNG shim with HTTP conditionals. The SQL layer should choose a transport family once from the URL scheme and then hand off to the appropriate adapter instead of scattering scheme-specific branches through session, wire, or Arrow code.

Because this transport-family split is now explicit, `http://` and `https://` URLs are reserved rather than silently passed into NNG operations. Current SQL surfaces fail fast on those schemes with a clear message that HTTP transport adapters are planned work, not current NNG endpoints. That behavior is deliberate: it keeps the transport boundary honest while future `ducknng_start_http_server(...)` and `ducknng_ncurl(...)` work is still being designed.
