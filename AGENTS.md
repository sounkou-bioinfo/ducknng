# AGENTS.md

This repository implements `ducknng`, a pure C DuckDB extension that exposes a DuckDB-backed SQL and RPC server over NNG using Arrow IPC and nanoarrow C for payload encoding and decoding. The project should be framed consistently as a transport-and-protocol server rather than as a collection of ad hoc helper functions or fake demo wire paths. The correct mental model is a DuckDB SQL server with a manifest-declared RPC surface, not a bag of one-off opcodes.

When working in this repository, keep the architecture split clear. NNG is the transport layer. The `ducknng` RPC envelope is the method layer. Arrow IPC through nanoarrow C is the tabular payload layer. DuckDB execution and result handling is the database layer. The public protocol must not be defined in terms of `nanonext`, R-specific helper code, or any single client implementation. Client examples are welcome, but they are examples of the protocol rather than the protocol itself. At the same time, the transport-facing SQL API should follow the `nanonext` pattern closely: transport is chosen by the URL scheme, not by a proliferation of transport-specific helper functions. `inproc://`, `ipc://`, `tcp://`, and later `tls+tcp://` should be accepted through the same operation-oriented entry points when the underlying runtime supports them.

The method registry and manifest are central. New methods should be described through descriptors, registered through a registry, and exported through the manifest. The codebase should support adding one method, removing one method, registering methods in bulk, unregistering bundles of methods, and generating the manifest from the current registry state. Avoid scattering protocol behavior across the codebase without a corresponding registry or manifest representation. The manifest is part of the public contract and should stay aligned with implementation.

The binding specifications for the project live in `docs/protocol.md`, `docs/manifest.md`, `docs/types.md`, `docs/security.md`, and `docs/registry.md`. Treat those files as checklists as well as design documents. Before implementing or changing a public method, make sure the transport pattern is defined, the payload format is declared, the method contract is present in the manifest model, the registry and descriptor contract are respected, the type mapping is covered by `docs/types.md`, and the concurrency, security, or session behavior is explicit. If a proposed change does not fit those documents, update the documents first and then implement. For subagent-style work in this repo, also use the project-local skills under `.pi/skills/` and keep `docs/subagents.md` aligned with the preferred decomposition and handoff workflow.

On the extension side, remain in C. Do not migrate core protocol or execution paths into DuckDB C++ APIs. Keep raw `nng_*` interactions behind `ducknng_nng_compat.c` and related headers. Keep version-sensitive DuckDB streaming behavior behind `ducknng_duckdb_streaming_compat.c`. Keep protocol encoding and decoding logic concentrated in the wire and IPC layers rather than leaking binary details into unrelated files. Do not reintroduce fake operations such as toy ping or hello flows simply to make tests pass. Transport autodetection, scheme validation, and TLS configuration belong in the NNG compatibility layer rather than in the SQL surface.

`ducknng` should now be treated explicitly as an unstable C API extension, not as a stable cross-version C API extension. The build metadata opts into `C_STRUCT_UNSTABLE`, and the repo targets one DuckDB version line at a time. This is intentional because the deeper Arrow/runtime direction depends on unstable ABI surface area that is not available through the stable extension ABI. That unstable stance is now justified in landed implementation, not only in metadata: the current row-bearing RPC export path uses DuckDB unstable Arrow conversion APIs directly through the extension surface, specifically `duckdb_result_get_arrow_options()`, `duckdb_to_arrow_schema()`, and `duckdb_data_chunk_to_arrow()`. Any docs or release notes that imply stable binary compatibility across multiple DuckDB versions should be corrected rather than softened.

Arrow IPC is the normal payload format for tabular input and output. Use nanoarrow C for encoding and decoding. JSON is appropriate for manifest and lightweight control metadata. Do not invent custom row serialization formats when Arrow IPC can express the payload. For row-bearing RPC export, prefer DuckDB-native Arrow conversion APIs when the unstable extension surface makes them available, and keep any remaining manual conversion seams clearly marked as transitional rather than architectural end state. Type claims must remain disciplined. Only advertise support for Arrow and DuckDB type mappings that are documented and tested. Favor a narrow, reliable supported subset over vague claims of broad compatibility.

The server should be designed for multiple clients. Stateless unary methods are fine where appropriate, but sessionful methods such as `query_open`, `fetch`, `close`, and `cancel` should make lifecycle and ownership explicit. Avoid hidden shared mutable state that blurs client boundaries. If a method is stateless, session-opening, session-requiring, or session-closing, that should be obvious in both the code and the manifest model.

Documentation style in this repository should stay direct and concrete. Prefer paragraphs over sprawling bullet lists unless a generated artifact specifically needs list formatting. Avoid planning-language fluff, fake phase theatre, and placeholder explanations that do not describe real behavior. README examples should run real code where possible and should describe client interoperability without implying that one client library defines the server protocol. The docs should be explicit that TLS material may come either from filesystem paths or from in-memory PEM content, and that the project needs helper utilities comparable to `nanonext` for generating self-signed development certificates and assembling client/server TLS configurations without forcing file-only workflows.

When adding or changing public behavior, also update tests, README material if user-visible behavior changed, and any generated metadata artifacts that depend on the public function or method surface. The aim is to keep `ducknng` coherent as a real SQL server over NNG with Arrow IPC, a predefined manifest, and an extensible method registry.
