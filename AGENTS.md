# AGENTS.md

This repository implements `ducknng`, a pure C DuckDB extension that exposes a DuckDB-backed SQL and RPC server over NNG using Arrow IPC and nanoarrow C for payload encoding and decoding. The project should be framed consistently as a transport-and-protocol server rather than as a collection of ad hoc helper functions or fake demo wire paths. The correct mental model is a DuckDB SQL server with a manifest-declared RPC surface, not a bag of one-off opcodes.

When working in this repository, keep the architecture split clear. NNG is the current transport adapter layer. The `ducknng` RPC envelope is the method layer. Arrow IPC through nanoarrow C is the tabular payload layer. DuckDB execution and result handling is the database layer. The public protocol must not be defined in terms of `nanonext`, R-specific helper code, or any single client implementation. Client examples are welcome, but they are examples of the protocol rather than the protocol itself. At the same time, the transport-facing SQL API should follow the `nanonext` pattern closely: transport is chosen by the URL scheme, not by a proliferation of transport-specific helper functions. `inproc://`, `ipc://`, `tcp://`, and `tls+tcp://` are the current NNG-facing schemes, and future `http://` / `https://` support should arrive as a separate transport adapter rather than as transport-specific copies of the RPC method surface. In that model, an aio is a future-like handle for one pending NNG operation backed by `nng_aio`, not a background job system or a new wire primitive. For `ducknng`, the first honest SQL-visible aio surface was raw request/reply futures such as `ducknng_request_raw_aio(...)` and `ducknng_request_socket_raw_aio(...)`, collected later through `ducknng_aio_collect(...)`. Generic raw `send_socket_raw_aio(...)` and `recv_socket_raw_aio(...)` helpers now also exist as transport primitives, but they do not redefine the session or RPC contract.

The method registry and manifest are central. New methods should be described through descriptors, registered through a registry, and exported through the manifest. The codebase should support adding one method, removing one method, registering methods in bulk, unregistering bundles of methods, and generating the manifest from the current registry state. Avoid scattering protocol behavior across the codebase without a corresponding registry or manifest representation. The manifest is part of the public contract and should stay aligned with implementation.

The binding specifications for the project live in `docs/protocol.md`, `docs/manifest.md`, `docs/types.md`, `docs/security.md`, `docs/registry.md`, `docs/transports.md`, and `docs/http.md`. Treat those files as checklists as well as design documents. Before implementing or changing a public method, make sure the transport pattern is defined, the payload format is declared, the method contract is present in the manifest model, the registry and descriptor contract are respected, the type mapping is covered by `docs/types.md`, and the concurrency, security, or session behavior is explicit. If a proposed change does not fit those documents, update the documents first and then implement. For subagent-style work in this repo, also use the project-local skills under `.pi/skills/` and keep `docs/subagents.md` aligned with the preferred decomposition and handoff workflow.

On the extension side, remain in C. Do not migrate core protocol or execution paths into DuckDB C++ APIs. Keep raw `nng_*` interactions behind `ducknng_nng_compat.c` and related headers. Keep transport-family parsing and URL-family decisions above adapter-specific code so a future HTTP adapter can reuse the same boundary without growing conditionals inside session or Arrow code. Keep protocol encoding and decoding logic concentrated in the wire and IPC layers rather than leaking binary details into unrelated files. Do not reintroduce fake operations such as toy ping or hello flows simply to make tests pass. Transport autodetection, scheme validation, and TLS configuration belong in dedicated transport code rather than in the SQL surface.

Keep DuckDB extension API usage deliberate. The current implementation avoids both unstable and deprecated DuckDB extension entrypoints, and any proposal to reintroduce them should be justified explicitly before landing it.

Arrow IPC is the normal payload format for tabular input and output. Use nanoarrow C for encoding and decoding. JSON is appropriate for manifest and lightweight control metadata. Do not invent custom row serialization formats when Arrow IPC can express the payload. The current row-bearing RPC export path uses explicit nanoarrow schema and batch mappings instead of DuckDB-specific helper conversions. Type claims must remain disciplined. Only advertise support for Arrow and DuckDB type mappings that are documented and tested. Favor a narrow, reliable supported subset over vague claims of broad compatibility.

The server should be designed for multiple clients. Stateless unary methods are fine where appropriate, but sessionful methods such as `query_open`, `fetch`, `close`, and `cancel` should make lifecycle and ownership explicit. Avoid hidden shared mutable state that blurs client boundaries. If a method is stateless, session-opening, session-requiring, or session-closing, that should be obvious in both the code and the manifest model.

Documentation style in this repository should stay direct and concrete. Prefer paragraphs over sprawling bullet lists unless a generated artifact specifically needs list formatting. Avoid planning-language fluff, fake phase theatre, and placeholder explanations that do not describe real behavior. README examples should run real code where possible and should describe client interoperability without implying that one client library defines the server protocol. The docs should be explicit that TLS material may come either from filesystem paths or from in-memory PEM content, and that the project needs helper utilities comparable to `nanonext` for generating self-signed development certificates and assembling client/server TLS configurations without forcing file-only workflows.

When adding or changing public behavior, also update tests, README material if user-visible behavior changed, and any generated metadata artifacts that depend on the public function or method surface. For every new user-visible function, result shape, or workflow slice, add a runnable `README.Rmd` illustration unless there is a concrete reason it cannot run in the rendered README; in that case, document the limitation explicitly rather than silently skipping the example. After rendering `README.Rmd`, always inspect the generated `README.md` as a reader would: check that the output is understandable, that long result blocks or generated material do not drown the important explanation, that paths are relative rather than machine-local, and that the prose clearly separates user-facing guidance from deeper protocol/design notes under `docs/`. Do not treat internal project checklists or design-review notes as substitutes for actual user documentation in the README. The aim is to keep `ducknng` coherent as a real SQL server over NNG with Arrow IPC, a predefined manifest, and an extensible method registry.
